# 1. 响应码体系不一致

**现状**：前端 `request.ts` 判断 `code === 200 || 201` 为成功；后端统一返回 `code: 0` 为成功。
**影响面**：所有接口——接上后端即全局不可用，每个请求都走 `message.error('请求失败')`。

- [x] **方案A（改前端）**：`request.ts` 中将判断改为 `data.code !== 0`。一行改动，后端零成本。**推荐。**
- [ ] **方案B（改后端）**：后端所有接口改为 `code: 200`。违反后端规范且需改大量测试用例。
- [ ] **方案C（中间适配）**：前端拦截器做 code 映射。增加魔法层，不推荐。

# 2. 登录响应结构不匹配

**现状**：前端期望 `{ token, admin: {id, username, avatar, role, role_name}, permissions: [...] }`；后端返回 `{ access_token, expires_in, admin: {id, username, role} }`。
**影响面**：登录后 token 存不上、`admin.avatar` 和 `admin.username` 为空。

- [x] **方案A（前端适配字段映射）**：`authStore.login()` 中做 `token ← access_token`，admin 字段按后端实际返回调整；permissions 由后端返回全权限数组（见 #8）。改动集中在一个文件。**推荐。**
- [ ] **方案B（后端改字段名）**：后端 `AdminLoginResponse` 改名为 `token`、加 `admin.role_name`。字段名适配前端习惯，但后端改 schema 影响测试用例。
- [ ] **方案C（混合）**：前端做字段映射 + 后端补齐缺少的 `permissions`、`avatar` 字段。各自动最小改动量。

### 具体改动

```
前端 authStore.login() 中：
  setToken(data.access_token)        // 原 data.token
  setAdminInfo(data.admin)           // 不变
  setPermissions(data.permissions)   // 后端补返回后即可用
```

# 3. URL 路径前缀不一致

**现状**：前端 `/admin/content/zones`、`/admin/content/courses`、`/admin/content/banners`；后端 `/admin/zones`、`/admin/courses`，**无** banner endpoint。
**影响面**：内容管理、课程管理、Banner 管理全部 404。

- [x] **方案A（改前端）**：前端 `services/content.ts` 中去掉 `/content` 前缀。banner 相关暂注释待后端补齐。**推荐。**
- [ ] **方案B（改后端）**：后端加一层 `/content` 前缀路由。两层路由多余。
- [ ] **方案C（nginx 重写）**：部署层做路径 rewrite。本地开发无效。

### 具体改动

| 文件 | 改动 |
|---|---|
| `src/services/content.ts` | `/admin/content/zones` → `/admin/zones`；`/admin/content/banners` → `/admin/banners`（暂注释）；`/admin/content/courses` → `/admin/courses` |
| `src/mock/content.ts` | Mock 拦截 URL 同步去前缀 |

# 4. 字段命名与类型不一致（前端全按后端来）

**决策**：后端 `admin/quiz.py` 和 `admin/orders.py` 复用了 C 端小程序的 schema（`QuizQuestionResponse`、`OrderResponse`），改后端 schema 会影响小程序。因此**前端完全按后端字段名来**，不动后端。

**现状**：后端输出 snake_case，部分字段名与前端不同（如 `sort_weight` ↔ `sort_order`），部分类型不同（如 `is_active: bool` ↔ `status: "online"/"offline"`）。

- [x] **方案A（前端全按后端来）**：前端类型定义 + 页面组件 + mock 数据全部改为后端字段名和类型。后端零改动，不受小程序 schema 共用约束。**推荐。**
- [ ] **方案B（axios 转换层）**：前端 `transformResponse` 做自动映射。增加全局黑盒逻辑，调试困难。
- [ ] **方案C（后端单独建 admin Response schema）**：订单和题库各加一个 Admin 版本 Response。干净但多两个文件 + 测试。

# 6. 题库导入方式不同

**现状**：前端粘贴 TXT → 解析后 POST JSON；后端接收 CSV 文件上传。
**影响面**：题库导入功能完全不可用。

- [x] **方案A（后端加 JSON 导入）**：后端新增接受 JSON body 的导入端点（保留 CSV 导入）。操作体验更好。**推荐。**
- [ ] **方案B（前端改文件上传）**：前端改为上传 CSV 文件。运营门槛高。
- [ ] **方案C（两种都支持）**：最灵活但维护成本高。

# 7. 筛选能力差异

**现状**：用户列表前端支持 username/phone/日期筛选，后端仅 openid/phone；订单列表前端支持 cert_type/phone 筛选，后端不支持；课程/内容列表前端支持 keyword 搜索，后端不支持。

- [x] **方案A（后端补齐筛选参数）**：各列表接口增加前端需要的 query 参数 + WHERE 条件。**推荐。**
- [ ] **方案B（前端本地筛选）**：大数据量时性能差、分页失效。
- [ ] **方案C（协商裁剪）**：对齐实际需要，减少无效开发。

### 筛选能力差异明细

| 接口 | 前端支持的筛选 | 后端当前支持的筛选 |
|---|---|---|
| `GET /admin/users` | username, phone, register_date_range | openid, phone |
| `GET /admin/orders` | status, date_range, cert_type, user_phone | status, start_time, end_time |
| `GET /admin/courses` | keyword, category | 无（仅分页） |
| `GET /admin/zones` | keyword | 无（仅分页） |

# 8. Token 与权限体系

**决策**：当前后台运营人员 1-3 人，不需要权限分级。后端登录时固定返回全权限数组，前端代码保留不动。未来如需分级再演进。

**现状**：前端做了完整的权限过滤体系（菜单 + 按钮级），后端 admin 使用角色枚举但无细粒度权限码。

- [x] **方案A（不做分级，全权限）**：后端 `AdminAuthService.login()` 中返回固定的全权限数组（包含前端路由 meta 中所有权限码）。前端零改动。**推荐。**
- [ ] **方案B（角色映射）**：后端按 role 返回不同权限数组。需要管理角色字典。
- [ ] **方案C（RBAC 表）**：新增 role_permissions 表。当前业务不需要。

### 具体实现

```python
# 后端 AdminAuthService.login() 返回时附带
ALL_PERMISSIONS = [
    "dashboard:view", "user:list",
    "order:list", "order:write",
    "quiz:list", "quiz:write", "quiz:import",
    "content:list", "content:write", "content:banner",
    "course:list", "course:write",
]
# data.permissions = ALL_PERMISSIONS
```
